Business Context

The Problem with “How Are We Doing?”

Every business asks it. Most cannot answer it precisely. The schema built across the previous eight chapters contains every transaction in the customer journey — but a row count is not a metric, and a metric is not a KPI. This chapter is about the gap between having data and having a number that means something.

The transactional tables in Tiers I–III are optimized for recording reality accurately. They are not optimized for answering the question “how are we doing this quarter compared to last?” reliably, quickly, or consistently across teams. Tier IV adds three tables that bridge that gap: the period table, the KPI definition table, and the KPI snapshot table. Together they turn a transactional schema into a measurement system.

Core Concept

KPI Design: Four Requirements

A well-designed KPI must satisfy all four of the following. If any one is missing, the number is a data point — not a KPI.

RequirementWhat It Means
Specific questionThe metric answers exactly one question, stated in plain language
Defined calculationNumerator and denominator are written down, not assumed
Time windowThe period is bounded and consistently applied
BaselineThere is a prior period or target to compare against
⚠ Watch Out — The Denominator Problem
The most common KPI failure is an undefined denominator. “On-time delivery rate” sounds simple until you ask: on-time out of what? All deliveries? Only completed ones? Only those with a promised date? Does a delivery with two shipments count once or twice?

Every denominator must be written down. If two people in the same meeting would calculate the same metric differently, the KPI is not defined.

Schema Additions

Tier IV: The Period Table

Calculating metrics from the transactional schema on demand is expensive and fragile. Tier IV adds three tables that make reporting fast, consistent, and auditable. The most foundational is the period table — deceptively simple, deeply leveraged.

ColumnAttributeNotes
per_idPeriod IDIV Primary key
per_namePeriod NameIV e.g., '2025-Q3', '2025-10', '2025-Annual'
per_typePeriod TypeIV Monthly · Quarterly · Annual · Custom
per_start_dateStart DateIV Inclusive
per_end_dateEnd DateIV Inclusive
per_prior_idPrior Period IDIV FK → period.per_id (self-reference); enables period-over-period joins in one line
per_is_completeIs CompleteIV Boolean; incomplete periods must be flagged in dashboards — never silently excluded
Why this matters more than it appears: Without a period table, date boundaries are scattered across hundreds of queries, BI tool fields, and spreadsheet formulas — each slightly different. The cost stays invisible until someone asks why the sales report shows Q3 revenue as $4.2M and the finance report shows $3.9M. The answer is almost always a boundary problem. A period table eliminates that class of error entirely. The boundary is defined once, in one place, by one owner.

The self-reference (per_prior_id): Prior-period comparison is always one join away — no hardcoded dates, no special-casing for year boundaries, no January vs. December arithmetic. The pattern is the same whether you are comparing months, quarters, fiscal years, or custom campaign windows.

Tier IV: KPI Definition & Snapshot Tables

The kpi_definition table solves the denominator problem structurally — by making the definition of every metric a row in the database, not tribal knowledge. The kpi_snapshot table freezes calculated values at a point in time so dashboards don’t drift when underlying data is updated mid-period.

Analytical Application

Queries That Become Possible

Open order count with prior period comparison

SELECT
  p.per_name AS period,
  ks.ks_value AS open_orders,
  ks_prior.ks_value AS prior_open_orders,
  ROUND((ks.ks_value - ks_prior.ks_value) / NULLIF(ks_prior.ks_value, 0) * 100, 1) AS pct_change
FROM kpi_snapshot ks
JOIN period p ON ks.ks_per_id = p.per_id
JOIN kpi_definition kd ON ks.ks_kpi_id = kd.kpi_id
LEFT JOIN period p_prior ON p.per_prior_id = p_prior.per_id
LEFT JOIN kpi_snapshot ks_prior
    ON ks_prior.ks_per_id = p_prior.per_id
    AND ks_prior.ks_kpi_id = ks.ks_kpi_id
    AND ks_prior.ks_segment IS NULL
WHERE kd.kpi_name = 'Open Order Count'
  AND ks.ks_segment IS NULL
  AND p.per_type = 'Monthly'
ORDER BY p.per_start_date DESC;

The per_prior_id self-join replaces all hardcoded date arithmetic. The same pattern works for any metric, any period type.

On-time delivery rate — rolling 13 months via period table

SELECT
  p.per_name,
  COUNT(CASE WHEN d.del_date <= o.ord_promised_date THEN 1 END) AS on_time,
  COUNT(*) AS total_delivered,
  ROUND(
    COUNT(CASE WHEN d.del_date <= o.ord_promised_date THEN 1 END) * 100.0
    / NULLIF(COUNT(*), 0), 1
  ) AS on_time_pct
FROM delivery d
JOIN "order" o ON d.ord_id = o.ord_id
JOIN period p ON d.del_date BETWEEN p.per_start_date AND p.per_end_date
WHERE d.del_status = 'Delivered'
  AND d.del_type != 'Internal-Transfer'
  AND p.per_type = 'Monthly'
  AND p.per_start_date >= CURRENT_DATE - INTERVAL '13 months'
GROUP BY p.per_id, p.per_name, p.per_start_date
ORDER BY p.per_start_date;

Period table join replaces hardcoded date ranges. Adding a carrier or segment breakdown requires only one additional GROUP BY column.

Dashboard seed — all six KPIs, current vs. prior period

SELECT
  kd.kpi_name, kd.kpi_unit,
  p.per_name AS current_period,
  ks.ks_value AS current_value,
  p_prior.per_name AS prior_period,
  ks_prior.ks_value AS prior_value,
  ROUND((ks.ks_value - ks_prior.ks_value) / NULLIF(ks_prior.ks_value, 0) * 100, 1) AS pct_change,
  p.per_is_complete AS period_complete
FROM kpi_snapshot ks
JOIN kpi_definition kd ON ks.ks_kpi_id = kd.kpi_id
JOIN period p ON ks.ks_per_id = p.per_id
LEFT JOIN period p_prior ON p.per_prior_id = p_prior.per_id
LEFT JOIN kpi_snapshot ks_prior
    ON ks_prior.ks_per_id = p_prior.per_id
    AND ks_prior.ks_kpi_id = ks.ks_kpi_id
    AND ks_prior.ks_segment IS NULL
WHERE ks.ks_segment IS NULL
  AND p.per_type = 'Monthly'
  AND p.per_end_date = (SELECT MAX(per_end_date) FROM period WHERE per_type = 'Monthly')
ORDER BY kd.kpi_name;

per_is_complete surfaces in the result so the presentation layer can flag incomplete periods — rather than silently comparing month-to-date to a full prior month.

Core Concept

Three Dashboard Rules & Three Period-Over-Period Errors

Dashboard Rules:
1. Every number needs a visible comparison — prior period, target, or both. A standalone metric has no meaning.
2. Incomplete periods must be flagged. Month-to-date vs. full prior month will always look worse. Use per_is_complete.
3. Segment before you summarize. An 87% on-time rate may hide one tier at 62% and another at 97%.
⚠ Period-Over-Period Errors
1. Comparing incomplete to complete periods — 19 days vs. 31 days almost always shows a false decline.
2. Shifting segment boundaries — if “enterprise” is redefined mid-year, the comparison is not the same population.
3. Ignoring mix shifts — overall on-time rate can improve while carrier performance degrades if volume shifts toward shorter routes. Summary metrics hide this; segment metrics surface it.
Reflection

Questions Worth Sitting With

  • Pick any metric your organization reports regularly. Write down its numerator and denominator in one sentence each. Is there ambiguity in either? Would two colleagues write the same thing?
  • Does your organization have a written definition of “new customer”? Does it distinguish a returning customer from a new one against full history — or only recent history?
  • Does your current reporting system flag incomplete periods, or are month-to-date figures routinely compared to full prior months without annotation?
  • Name one metric in your organization that shifted favorably while the business outcome it was meant to represent stayed flat or got worse. What mix shift or boundary change explains the gap?
  • If the six baseline metrics were calculated for your organization tomorrow, which one would produce the most disagreement — and what would that disagreement reveal about how definitions are managed?
