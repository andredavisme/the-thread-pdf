Core Concept

What a Sales Intelligence System Is

A dashboard is a presentation layer. A sales intelligence system is the infrastructure beneath it: the schema that stores transactions, the definitions that govern how those transactions are measured, the processes that keep both current, and the interfaces that put the right information in front of the right people at the right time.

A report tells you what happened. A system tells you what is happening and flags what needs attention. The difference is automation, alerts, and freshness — and all three depend on how the schema is populated and maintained.
System Design

Four Layers

LayerPurposeSchema Dependency
1. Data CaptureWrite accurate, complete records at the moment of the transactionStatus fields, controlled vocabularies, system-generated timestamps
2. AutomationTrigger status transitions, generate action queues, schedule KPI snapshotsord_status, del_status, adj_status, per_is_complete, resp_followup_required
3. CRM IntegrationPresent the schema through a user-facing application; write back through controlled interfacesAll Tier I–III tables (reads); request, quote, adjustment, response (writes)
4. ReportingSurface KPI snapshots and period-over-period comparisons to decision-makerskpi_snapshot, period, kpi_definition
⚠ Layer 1 Is the Most Common Failure Point
The schema is only as good as the data written into it. Status fields should update when the triggering event occurs, not when someone remembers. Controlled vocabularies (adj_type, resp_sentiment, resp_category) should be enforced at the application layer — free-text alternatives are a data quality tax paid on every query. Timestamps should be system-generated, not user-entered.

Core Concept

The Living Schema

No schema survives first contact with production unchanged. Three practices separate schemas that remain useful over time from those that accumulate technical debt until they are replaced.

Practice 1 — Changes flow through kpi_definition
When a KPI definition changes, record the change in kpi_definition before applying it to snapshot calculations. Update kpi_last_reviewed, annotate what changed in the description fields. Historical snapshots are preserved under the prior definition. New snapshots are calculated under the new one. Without this, a trend line that appears to show improvement may actually reflect a redefinition.
Practice 2 — New fields are additive, not overwriting
A field no longer populated is deprecated with a note, not deleted — historical records contain it. A table replaced by a more structured version is retained, not dropped. The cost of keeping deprecated fields is low. The cost of losing historical continuity is high. This is the same principle the tier system used throughout this textbook: add, don’t replace.
Practice 3 — Definitions are owned, not assumed
Every kpi_definition row has a kpi_owner. Every period definition is loaded by a named process. Every schema change is attributable to a decision. This is not bureaucracy — it is the difference between a schema that is auditable and one that is mysterious. When a number changes unexpectedly, the question is always: was this a change in the business, or a change in how we measured it? A schema with ownership can answer that. One without it cannot.

Analytical Application

Full Customer Intelligence View — Last 12 Months

Assembles order volume, realized margin, on-time rate, adjustment count, and NPS for every active customer in a single query.

WITH customer_orders AS (
  SELECT o.ord_cust_id,
    COUNT(DISTINCT o.ord_id) AS total_orders,
    SUM(ol.ord_ln_unit_price * ol.ord_ln_qty_fulfilled) AS gross_revenue,
    SUM(s.sku_cost * ol.ord_ln_qty_fulfilled) AS gross_cost
  FROM "order" o
  JOIN order_line ol ON o.ord_id = ol.ord_id
  JOIN sku s ON ol.sku_id = s.sku_id
  WHERE o.ord_date >= CURRENT_DATE - INTERVAL '12 months'
  GROUP BY o.ord_cust_id
),
adjustment_impact AS (
  SELECT o.ord_cust_id,
    COUNT(DISTINCT a.adj_id) AS total_adjustments,
    SUM(COALESCE(a.adj_price_delta, 0)) AS total_price_adj
  FROM adjustment a
  JOIN "order" o ON a.adj_ord_id = o.ord_id
  WHERE a.adj_status = 'Applied'
    AND o.ord_date >= CURRENT_DATE - INTERVAL '12 months'
  GROUP BY o.ord_cust_id
),
delivery_performance AS (
  SELECT o.ord_cust_id,
    COUNT(d.del_id) AS total_deliveries,
    COUNT(CASE WHEN d.del_actual_date <= d.del_promised_date THEN 1 END) AS on_time
  FROM delivery d JOIN "order" o ON d.del_ord_id = o.ord_id
  WHERE d.del_status = 'Delivered' AND d.del_type != 'Internal-Transfer'
    AND o.ord_date >= CURRENT_DATE - INTERVAL '12 months'
  GROUP BY o.ord_cust_id
),
nps_summary AS (
  SELECT r.cust_id, ROUND(AVG(r.resp_nps), 1) AS avg_nps, COUNT(r.resp_id) AS response_count
  FROM response r
  WHERE r.resp_nps IS NOT NULL AND r.resp_date >= CURRENT_DATE - INTERVAL '12 months'
  GROUP BY r.cust_id
)
SELECT c.cust_id, c.cust_name, c.cust_segment,
  COALESCE(co.total_orders, 0) AS orders_12m,
  ROUND(COALESCE(co.gross_revenue,0) + COALESCE(ai.total_price_adj,0), 2) AS realized_revenue,
  ROUND((COALESCE(co.gross_revenue,0)+COALESCE(ai.total_price_adj,0)-COALESCE(co.gross_cost,0))
    / NULLIF(COALESCE(co.gross_revenue,0)+COALESCE(ai.total_price_adj,0),0)*100, 1) AS margin_pct,
  ROUND(COALESCE(dp.on_time,0)*100.0/NULLIF(dp.total_deliveries,0),1) AS on_time_pct,
  COALESCE(ai.total_adjustments,0) AS adjustments_12m,
  ns.avg_nps, ns.response_count
FROM customer c
LEFT JOIN customer_orders co ON c.cust_id = co.ord_cust_id
LEFT JOIN adjustment_impact ai ON c.cust_id = ai.ord_cust_id
LEFT JOIN delivery_performance dp ON c.cust_id = dp.ord_cust_id
LEFT JOIN nps_summary ns ON c.cust_id = ns.cust_id
WHERE co.total_orders > 0
ORDER BY realized_revenue DESC;

Five CTEs, one final SELECT. Every signal from the full customer journey assembled into a single customer intelligence row, ordered by realized revenue.

Open action queue — expiring quotes, pending adjustments, open follow-ups in one view

SELECT 'Quote Expiring' AS action_type, c.cust_name, q.qte_id AS record_id,
  q.qte_valid_until AS due_date,
  (julianday(q.qte_valid_until) - julianday('now')) AS days_remaining,
  u.user_first_name || ' ' || u.user_last_name AS owner
FROM quote q JOIN customer c ON q.cust_id = c.cust_id
LEFT JOIN "user" u ON q.qte_prepared_by = u.user_id
WHERE q.qte_status NOT IN ('Accepted','Cancelled','Expired-No-Action','Superseded')
  AND q.qte_valid_until <= CURRENT_DATE + INTERVAL '14 days'
UNION ALL
SELECT 'Adjustment Pending Approval', c.cust_name, a.adj_id,
  a.adj_initiated_date + INTERVAL '5 days',
  (julianday('now') - julianday(a.adj_initiated_date)), u.user_first_name || ' ' || u.user_last_name
FROM adjustment a JOIN "order" o ON a.adj_ord_id = o.ord_id
JOIN customer c ON o.ord_cust_id = c.cust_id
LEFT JOIN "user" u ON a.adj_approved_by = u.user_id
WHERE a.adj_status = 'Pending'
UNION ALL
SELECT 'Response Follow-Up Due', c.cust_name, r.resp_id,
  r.resp_date, (julianday('now') - julianday(r.resp_date)),
  u.user_first_name || ' ' || u.user_last_name
FROM response r JOIN customer c ON r.cust_id = c.cust_id
LEFT JOIN "user" u ON r.resp_followup_owner = u.user_id
WHERE r.resp_followup_required = TRUE AND r.resp_resolved = FALSE
ORDER BY days_remaining DESC;

System health check — data quality and snapshot freshness

SELECT 'Orders with NULL status' AS check_name, COUNT(*) AS issue_count FROM "order" WHERE ord_status IS NULL
UNION ALL
SELECT 'Deliveries with NULL promised date', COUNT(*) FROM delivery
  WHERE del_promised_date IS NULL AND del_status != 'Cancelled'
UNION ALL
SELECT 'Applied adjustments with no root cause', COUNT(*) FROM adjustment
  WHERE adj_root_cause IS NULL AND adj_status = 'Applied'
UNION ALL
SELECT 'Responses: score but no scale', COUNT(*) FROM response
  WHERE resp_score IS NOT NULL AND resp_score_scale IS NULL
UNION ALL
SELECT 'KPI snapshots older than 48h (complete periods)', COUNT(*)
FROM kpi_snapshot ks JOIN period p ON ks.ks_per_id = p.per_id
WHERE p.per_is_complete = TRUE AND ks.ks_calculated_at < NOW() - INTERVAL '48 hours'
ORDER BY issue_count DESC;

Run this weekly. Any non-zero count is a defined question: a data entry gap, an automation failure, or a snapshot process that didn’t run.

Reflection

Questions Worth Sitting With

  • Of the four system layers — data capture, automation, CRM integration, and reporting — which one is most likely to be the weakest in your organization today? What would break if you audited it honestly?
  • Does your organization currently have a single view across expiring quotes, pending adjustments, and open follow-ups? Or are those three lists maintained separately, by different teams, in different tools?
  • The journey completeness audit surfaces requests with no estimate, estimates with no quote, orders with no delivery. How many records in your current system would that query return? What does the answer tell you about where the customer journey breaks down most often?
  • The living schema practice of preserving deprecated fields rather than deleting them has a cost: a schema that accumulates columns over time. Where does your organization draw that line, and how is the decision made?
  • If the system health check query ran against your database tomorrow, how confident are you in what those counts would be? The answer is a measure of how well you know your own data.
